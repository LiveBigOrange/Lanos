## 一、架构定义：Go ↔ Flutter 通信方案

### 1.1 决策：本地 HTTP API（Go 作为 HTTP 服务端）

```
┌──────────────────┐      HTTP REST API      ┌──────────────────────┐
│  Flutter UI       │ ◄──────────────────────► │  Go Core Daemon      │
│  (桌面图形界面)     │     localhost: 随机端口   │  (mDNS/传输/分享服务)  │
│                  │       JSON over HTTP     │                      │
│  进程 B           │                          │  进程 A               │
└──────────────────┘                          └──────────────────────┘
```

**理由**（对比其他方案）：

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| HTTP API（选此方案） | 解耦彻底，Go 崩溃不影响 UI，可独立调试、热重启 Flutter；支持第三方客户端 | 环回 HTTP 有轻微开销，需管理服务生命周期 | **MVP 选型** |
| cgo/FFI 编译为共享库 | 无进程间通信开销，单进程部署 | 三端交叉编译复杂，任一语言 panic 导致整个进程崩溃；Flutter 桌面端 cgo 集成缺乏成熟社区实践 | 暂不采用 |
| Unix Socket / stdin/stdout | 比 HTTP 更轻量 | 跨平台差异大（Windows 不支持 Unix Socket）；stdin/stdout 无请求路由，结构化程度低 | 不采用 |

### 1.2 生命周期管理

- **Go Core Daemon（gcd）**：作为一个独立后台进程运行
- **启动方式**：Flutter 启动时通过 `Process.start` 自动拉起 gcd；若已运行则复用
- **守护策略**：gcd 自身不做 daemon 化，由 UI 进程管理其生命周期
- **退出处理**：Flutter 窗口关闭时，可选择（a）保留 gcd 后台运行（托盘模式）或（b）发送 `POST /shutdown` 优雅退出
- **健康检查**：Flutter 定期（每 5 秒）`GET /ping`，连续 3 次无响应则重启 gcd

### 1.3 API 设计原则

- RESTful，`/api/v1/` 前缀
- 请求/响应均为 JSON
- 关键事件（文件接收状态、传输进度）通过 Server-Sent Events（SSE）`GET /api/v1/events` 推送，Flutter 用 `EventSource` 接收
- SSE 事件类型：
  - `device.online` / `device.offline`
  - `transfer.request`（接收方收到发送请求）
  - `transfer.progress`（进度更新）
  - `transfer.complete` / `transfer.failed`
  - `share.downloaded` / `share.expired`

## 二、传输框架补充

### 2.1 传输队列与并发策略

- **队列模型**：每个目标设备维护独立传输队列，设备间传输并行
- **单设备串行**：同一时刻向同一设备只传一个任务，其余排队（避免接收方 UI 过载）
- **优先级**：网页分享的 HTTP 下载 > 用户拖拽发起的直传（确保访客下载体验顺畅）
- **取消操作**：发送方和接收方均可取消进行中的传输，取消后发送方清理临时数据

### 2.2 传输状态机

```
待确认（发送方等待接收方确认码确认）
  │
  ▼
连接中（加密通道握手）
  │
  ▼
传输中（分块发送 + 进度回调）
  │
  ├──► 已完成
  ├──► 已取消（任意方主动取消）
  ├──► 已失败（网络断开 / 超时）
  └──► 待续传（失败后可手动重试，保留已接收片段）
```

### 2.3 续传设计（MVP 简化版）

- **分块大小**：固定 4MB 一个 chunk
- **进度记录**：接收方每收到一个 chunk 写入 `.transfer_cache/<task_id>/` 临时目录，记录已完成的 chunk 索引列表到 `.transfer_cache/<task_id>.meta`
- **重试机制**：失败后 UI 显示"重试"按钮；重试时发送方读取 meta 文件跳过已完成的 chunks
- **边界**：MVP 仅支持手动重试（不自动重连），且接收方未点击"接收"前不提前建立连接

### 2.4 并发连接数限制

- 最大同时处理传输任务数：4（上行）+ 4（下行），防止大量文件请求耗尽系统资源
- 网页分享同时下载连接数：单个分享最多 3 个并发 HTTP 连接

## 三、网页分享架构补充

### 3.1 服务模型：单一 HTTP 服务路由所有 token

```
一个 HTTP(S) 服务监听一个端口，按路径路由：
  /dl/<token>          — 文件下载（校验 token + 可选密码）
  /api/share/<token>/status — 返回文件名、大小、剩余次数/时间（JSON）
  /qr/<token>          — 二维码页面（HTML）
```

**不采用**"每个链接启动一个独立服务"的方案，原因：
- 端口资源有限
- 用户无法同时创建大量分享（64+）
- 资源清理复杂

### 3.2 同时分享上限

- 默认同时活跃分享上限：64 个
- 达到上限后创建新分享时提示："已达同时分享上限（64），请先停止一些分享后再试"
- 设置中可调整此上限（建议范围：16-256）

### 3.3 大文件夹 ZIP 流式处理

- **方案**：使用 Go 的 `archive/zip` + `io.Pipe`，边压缩边发送
- **限制**：分享文件夹时，若总大小超过 4GB，提示用户"文件夹超过 4GB，部分浏览器可能无法处理"但允许继续
- **超时**：单次 ZIP 流式传输超过 30 分钟自动断开，避免长时间占用资源
- **中文文件名**：ZIP 条目使用 `utf-8` + `language encoding flag (0x0800)` 确保跨平台解压正常

### 3.4 HTML 下载页

当浏览器访问下载链接时，返回一个极简 HTML 页面：
- 文件名 + 文件大小（文件夹显示"ZIP 压缩包，共 N 个文件"）
- 密码输入框（如有设密码）
- 下载按钮
- 进度提示
- 剩余下载次数 / 有效时间倒计时

## 四、安全与加密协议细化

### 4.1 直传加密：选择 Noise Protocol

**选择 Noise XK 模式**（而非 TLS），理由：

| 维度 | TLS 自签证书 | Noise XK |
|------|-------------|----------|
| 首次连接 | 需要用户确认证书指纹（易与"确认码"概念混淆） | 确认码直接嵌入密钥确认机制 |
| 握手开销 | 2-RTT + Certificate 开销 | 1-RTT，无证书 |
| 依赖 | 需要 x509 证书生成/管理 | 仅需 ed25519 公私钥对 |
| 浏览器兼容 | 网页分享需要 TLS 则需 Let's Encrypt | 直传与网页分享独立处理，各用各的 |

**实际策略**：
- **App ↔ App 直传**：使用 Noise XK + ed25519 密钥对，4 位确认码采用 Short Authentication String (SAS) 机制验证首次连接
- **网页分享 HTTPS**：自签证书 + 用户首次访问时浏览器提示风险（无可避免），或 MVP 阶段直接使用 HTTP + token 安全（局域网 + 随机 token 已具备足够安全保障）

### 4.2 4 位确认码碰撞风险评估

| 网络规模 | 碰撞概率（生日攻击） | 风险等级 | 应对 |
|----------|-------------------|----------|------|
| 2-10 台设备 | ~0.01% | 极低 | 接受 |
| 50 台设备 | ~11.8% | 中等 | 一个 30 秒内有两个设备同时请求传输时会有碰撞 |
| 200+ 台设备 | ~86.5% | 高 | 建议企业场景升级到 6 位数 |

**MVP 决定**：维持 4 位数，在确认码界面加上**发送方设备名 + 接收方设备名**双重核对（用户不仅要看数字，还需确认对方设备名是否匹配），大幅降低实际碰撞风险。

### 4.3 密钥存储

- 每台设备首次启动时生成 ed25519 密钥对
- 私钥存储路径：
  - Windows: `%APPDATA%/Lanos/identity.key`（DPAPI 加密）
  - macOS: `~/Library/Application Support/Lanos/identity.key`（Keychain 封装）
  - Linux: `~/.config/lanos/identity.key`（0600 权限）
- 公钥通过 mDNS TXT record 广播，用于标识设备身份

## 五、防火墙与系统集成

### 5.1 平台策略

| 平台 | 端口放行策略 | 代码实现 |
|------|-------------|----------|
| Windows | 首次启动时通过 `netsh firewall` 或 WinFW API 自动添加放行规则（TCP + mDNS） | CGO 调用 `FirewallAddApp` 或 shell `netsh advfirewall` |
| macOS | 系统会自动弹窗询问，用户点"允许"即可 | Go 的 `ipfw`/`pfctl` 不再需要 |
| Linux | 检测 `ufw` / `firewalld` / `iptables`，提示用户手动放行或自动执行 | 提供 `lanos-setup-firewall.sh` 脚本 |
| **所有平台** | 网页分享端口范围（52000-53000）需要在防火墙中开放 | 设置向导中引导 |

### 5.2 Linux 系统托盘兼容

- 优先使用 `libappindicator`（Ubuntu/Debian 传统方案）
- 检测到 GNOME 40+（无 AppIndicator 支持）时，回退到：
  - 安装 `gnome-shell-extension-appindicator` 的提示
  - 或保持 Flutter 窗口常驻最小化

## 六、单实例与多开控制

- **强制单实例**：通过文件锁实现
  - 启动时尝试创建 `%TEMP%/lanos.lock`（平台对应临时目录）
  - 如果锁已存在且持有进程存活，弹出提示"Lanos 已在运行"并退出
- **允许场景**：不同的用户账户下可同时运行实例（互不干扰）

## 七、更新机制

- **MVP 方案**：手动检查更新
  - 设置页底部显示当前版本号 + "检查更新"按钮
  - 点击后请求 GitHub Releases API（`api.github.com/repos/xxx/lanos/releases/latest`）
  - 发现新版本时，打开浏览器跳转到下载页
- **暂不实现**：自动下载安装、增量更新、Windows `squirrel` 等自动更新框架（V2 考虑）

## 八、体验细节补充

### 8.1 文件选择 fallback

主界面除拖拽外，提供以下入口：
- **"+" 按钮**：点击后弹出系统文件选择对话框
- **"发送"按钮**：选择文件后弹出设备选择列表（或者贴在主界面做常驻快捷入口，接收方列表占用主界面的主要区域）
- **剪贴板文本快捷发送**：右键托盘 → "发送剪贴板文本"，自动将文本保存为临时 `.txt` 文件进入发送流程

### 8.2 通知粒度（MVP 简化版）

设置中统一控制三个开关：
- ☑ 接收文件通知（有新文件收到时弹通知）
- ☑ 传输完成通知（自己发的文件发送完成时弹通知）
- ☑ 链接状态通知（分享过期 / 被下载时弹通知）

不提供子分类（每条开关对应一组场景，不对应单个文件）。

### 8.3 记录管理增强

- **搜索**：记录列表上方搜索框，按文件名模糊匹配
- **批量操作**：支持多选 → 批量删除
- **排序**：按时间 / 文件大小 / 设备名排序
- **导出记录**：设置页提供"导出传输记录为 CSV"

### 8.4 语言切换

设置页 → "语言"下拉选项：
- 简体中文
- English

切换后立即生效（不重启），使用 Flutter `flutter_localizations` + 运行时 `setLocale`。

### 8.5 首次引导 3 步页

**Step 1**：欢迎 → "Lanos 让局域网传文件像 AirDrop 一样简单"
**Step 2**：功能介绍 → 三张插图卡片（发现设备 / 拖拽发送 / 生成链接）
**Step 3**：设置 → 设备名 + 下载路径 + 自动接收策略 + 开机自启

引导仅在首次安装时展示，可在设置中"重新查看引导"。

## 九、非功能指标补充

### 9.1 性能基准（千兆以太网）

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 单文件传输速度 | ≥ 80 MB/s | 1GB 文件，TCP 吞吐量（不包含 UI 渲染开销） |
| 100 个文件批量传输速度 | ≥ 40 MB/s | 100 个 10MB 文件，含握手 + 加密 |
| 内存占用（空闲） | ≤ 40 MB | go 进程 rss + flutter 进程 rss |
| 内存占用（传输中） | ≤ 120 MB | 传输 10GB 文件期间峰值 |
| CPU 占用（传输中） | ≤ 1 核 | ed25519 + chacha20 软件实现 |
| 网页分享首次响应 | ≤ 500ms | 从 HTTP 请求到首字节 |
| 设备发现延迟 | ≤ 3s | 从启动到列表中看到同局域网已在线设备 |

### 9.2 可靠性目标

- 连续传输 100 个文件零失败
- 传输过程中的 Flutter UI 不卡顿（帧率 ≥ 30fps）
- 单次会话（不重启）连续运行 7 天不崩溃

## 十、MVP 边界补充

### 10.1 明确包含

- ✅ Go Core 作为独立进程管理
- ✅ 单实例锁
- ✅ 防火墙自动/半自动配置
- ✅ 支持 IPv4 局域网（IPv6 暂不支持）
- ✅ 手动检查更新
- ✅ 记录搜索 + 批量删除
- ✅ 语言切换（简中/英文）
- ✅ 通知三开关

### 10.2 明确不包含

- ❌ 自动更新 / delta 更新
- ❌ IPv6 发现与传输
- ❌ 自动续传（MVP 仅手动重试）
- ❌ Linux AppImage（MVP 仅 Windows exe + macOS dmg，Linux 作为快速跟进）
- ❌ 剪贴板文本直接分享（可通过发送 .txt 文件变通）
- ❌ 网页分享 HTTPS（HTTP + token 安全在局域网环境中可接受）
- ❌ cgo / FFI 方案

## 十一、数据目录结构

```
用户数据根目录：
  Windows: %APPDATA%/Lanos/
  macOS:   ~/Library/Application Support/Lanos/
  Linux:   ~/.config/lanos/

├── identity.key         # ed25519 私钥
├── config.yaml          # 用户设置（设备名、下载路径、策略等）
├── trusted_devices.json # 可信设备列表（公钥 → 设备名 map）
├── transfer_cache/      # 正在接收的临时文件片段
│   └── <task_id>/
│       ├── chunk_00001
│       ├── chunk_00002
│       └── ...
├── transfer_meta/       # 传输元数据（续传状态）
│   └── <task_id>.meta
├── transfer_log.db      # 共享记录 + 接收记录（SQLite）
└── logs/                # 调试日志（脱敏，不含文件名）
    └── lanos_20260721.log
```
